源码深度解析

我读完了 OpenClaw 的 46 万行源码，发现了这些工程决策

作者：干活验证虾 × Benjamin 日期：2026.04.06 源码版本：OpenClaw 2026.4.4（commit: 4954d02） 代码规模：约 460,000 行 TypeScript

一架构全景：它不是「多渠道聊天机器人」

在读源码之前，我以为 OpenClaw 就是一个把 AI 对话桥接到多个聊天 app 的中间件。读完之后发现这个判断低估了它的工程规模。

OpenClaw 的本质是一个常驻 daemon 进程，围绕 ACP（Agent Control Protocol）构建的多租户 AI Agent 基础设施。它需要管理的不只是消息路由，而是：多 session 并发隔离、ACP harness 连接池、认证轮转、向量记忆、定时调度、子 agent 生命周期、安全审计……

这个架构的核心设计意图是：把 AI Agent 的基础设施问题（渠道适配、并发控制、资源管理、安全审计）彻底抽象出来，让 skill 和 plugin 开发者只关注业务逻辑。

二渠道层：一个接口适配 21+ 渠道的设计哲学

2.1 ChannelPlugin：四大适配器接口

每个渠道插件实现四个接口，其中两个是可选的。这个设计让渠道插件可以按能力「自报家门」：

interface ChannelPlugin {
  id: ChatChannelId;                    // 动态字符串（非枚举！）
  capabilities: ChannelCapabilities;   // 能力声明
  messaging: ChannelMessagingAdapter;  // 消息发送（必须）
  threading?: ChannelThreadingAdapter; // 线程管理（可选）
  security?: ChannelSecurityAdapter;   // 安全策略（可选）
}
工程决策

id 是动态字符串而不是枚举。原因是 OpenClaw 支持第三方渠道插件——如果用枚举，每新增一个渠道都需要修改核心类型定义；用字符串，插件只需注册自己的 id，核心代码不需要改。这是「开放-封闭原则」在渠道设计上的体现。

ChannelCapabilities 是渠道能力的完整声明清单：

能力字段说明支持示例
canReactemoji 反应Discord / Slack / Telegram
hasNativeThreading原生线程/话题Discord / Slack
supportsStreamingMessages流式输出（实时更新消息）Discord / Telegram
inlineButtons内嵌按钮（dm/group/all/allowlist）Telegram / Discord
canSendEphemeral临时消息（只有发送者可见）Slack
canSendPolls投票消息Telegram / Discord

2.2 StatusReactionController：状态 emoji 的工程精度

每次 AI 处理消息时，渠道上会有一个 emoji 实时反映当前状态。这个看似简单的功能背后有精确的工程设计：

// 状态机：queued → thinking → [tool 执行] → done/error
//                         ↘ coding/web（专用状态）
//                         ↘ stallSoft(10s) → stallHard(30s)
//                         ↘ compacting（压缩进行中）

// 中间状态 debounce 700ms —— 避免 thinking→tool→done 快速切换
scheduleEmoji(emoji, 700ms delay)

// 终止状态立即执行，不等 debounce
if (isFinalState) executeImmediately()
工程洞察

700ms debounce 不是随意选的。它比人眼感知阈值（约 100-200ms）大得多，目的不是「平滑 UI」，而是「减少 API 调用次数」。如果 thinking → tool 在 500ms 内完成，用户只会看到一个 emoji 变化，而不是两个。但 stallSoft（10s 无输出触发卡住检测）需要 Immediate 执行——用户需要知道 AI 可能卡住了。

2.3 DraftStreamLoop：流式 typing 效果的设计

AI 回复时，消息会实时更新（类似「正在输入」效果）。这个功能的并发控制很有讲究：

// 核心：不用队列，用覆盖
pendingText = text  // 永远是最新内容

// inFlightPromise：同一时刻只有一个 sendOrEditStreamMessage 在飞
// throttleMs：控制更新频率（避免触发渠道 API 频率限制）

update(text) → 更新 pendingText
  ├─ 有 inFlight → schedule（等 throttleMs）
  ├─ 超过 throttleMs → 立即 flush
  └─ 否则 → schedule

flush() → 发送 pendingText（不是队列里的历史消息）
工程决策

覆盖而非队列。流式输出的语义是「展示最新进度」，不是「保证每个中间状态都展示出来」。用队列会导致发送积压——用户看到的是「上上上个版本的内容」，而且越积越多。用覆盖，保证用户永远看到最新内容，同时 throttleMs 控制 API 调用频率。

2.4 inbound-debounce：防止连续输入触发多次

当用户快速发送多条消息时，防止每条都触发一次完整的 AI 对话：

  • 有 media 附件 → 不 debounce（立即处理，避免附件内存积压）
  • 是控制命令（/help、/status 等）→ 不 debounce（命令需要即时响应）
  • 普通文本 → 防抖（合并短时间内的多条消息）
  • maxTrackedKeys = 2048：防止 DDoS 场景下 Map 无限增长

三ACP 控制平面：1997 行的并发管理核心

OpenClaw 的 ACP manager（manager.core.ts，1997 行）是整个系统最复杂的模块。它管理着与 ACP harness（Claude Code、Codex、Cursor 等）的所有连接，以及多 session 的并发隔离。

3.1 SessionActorQueue：同 session 严格串行

class SessionActorQueue {
  // 核心原语：同一 actorKey 的操作严格串行
  async run(actorKey: string, op: () => Promise): Promise

  // 可监控字段（可接入 Prometheus 等）
  getPendingCountForSession(key: string): number  // 单 session 等待深度
  getTotalPendingCount(): number                   // 全局等待总数
}

// 使用方式
withSessionActor(sessionKey, async () => {
  // 新 turn 必须等上一个 turn 完成
  // 不允许同一 session 并发执行两个 turn
})
为什么需要串行化？

AI 对话是有状态的——第二条消息的上下文依赖第一条消息的完整处理结果。如果两个 turn 并发执行，状态会混乱（类似数据库的并发写入问题）。SessionActorQueue 是这个场景下最简单有效的解法：每个 session 一把锁，不同 session 互不干扰。

3.2 RuntimeCache：连接池 + LRU 驱逐

维护与 ACP harness 的长连接，用 LRU 策略驱逐空闲连接：

class RuntimeCache {
  // touch=true → 更新 lastTouchedAt（LRU 计数）
  // touch=false（peek）→ 不更新，用于只读检查
  get(actorKey, { touch?: boolean })
  
  // 收集空闲超过 maxIdleMs 的候选（不立即驱逐）
  collectIdleCandidates({ maxIdleMs })
  
  // 点快照（含 idleMs）→ 接入 observability
  snapshot()
}

// 关键字段
interface CachedRuntimeState {
  runtime: AcpRuntime;
  handle: AcpRuntimeHandle;
  appliedControlSignature?: string;  // undefined → 下次 turn 强制重推 controls
}
工程决策

appliedControlSignature 的作用。当 controls（系统提示、工具列表等配置）发生变化时，不是立即断开重连，而是把 signature 标记为 undefined，等下次 turn 开始时「顺带」重推 controls。这避免了配置更改导致的连接抖动——不需要建新连接，只需要在现有连接上发送更新的配置。

3.3 超时清理的双阶段设计

// 超时发生后：
// 阶段1：发中断信号，给 runtime 宽限期自行清理
abortController.abort()
runtime.cancel()
await awaitCleanupWithGrace(GRACE_MS)
// 超时 → 异步继续等（不阻塞），记录日志

// 仅 oneshot 模式：
// 阶段2：强制关闭连接
runtime.close()
await awaitCleanupWithGrace(GRACE_MS)
// 两阶段都成功 → 同步清除缓存
// 任一失败 → Promise.allSettled 后异步清除（不丢弃！）
工程洞察

两阶段超时清理防止资源泄漏。单阶段 abort 可能无法优雅停止正在运行的工具（比如一个长时间的 bash 命令）。两阶段设计给 runtime 机会自行完成清理，只有宽限期结束后才强制关闭。Promise.allSettled 而不是 Promise.all，确保即使某一步失败，清除操作也一定会执行。

3.4 ACP approval-classifier 反注入设计

当工具调用需要权限审批时，OpenClaw 从三个来源提取工具名：

// 三源一致性检查
const source1 = _meta.toolName        // 来自 ACP 元数据
const source2 = rawInput.tool         // 来自工具调用参数
const source3 = title.split(':')[0]   // 来自显示标题

// 任何两源有值但不一致 → undefined（拒绝）
// 防止 title 被篡改来绕过权限检查
if (source1 && source2 && source1 !== source2) return undefined
工程决策

三源一致性是反注入的关键。如果只从 title 提取工具名，攻击者可以构造 bash: [actually safe tool name] 来伪装 bash 命令为安全工具。通过比对三个来源，任何不一致都会被识别为潜在的注入尝试，返回 undefined 触发权限拒绝。

四Agent 运行时：2098 行的对话循环

4.1 eventGate：防超时污染的优雅方案

这是我在 OpenClaw 源码里看到的最优雅的设计之一：

const eventGate = { open: true }

// turn 超时后：关闭 gate
// ⚠️ 不停止 stream（停止 stream 可能导致 runtime 状态不一致）
eventGate.open = false

// 事件仍从 stream 流出，但被 gate 过滤
for await (const event of pi.runTurn()) {
  if (!eventGate.open) continue  // 超时后的事件全部忽略
  
  // 正常处理
  handleEvent(event)
}
为什么不直接 break？

如果 turn 超时后直接 break 退出循环，stream 还在运行中，后续的事件（可能包含工具结果、状态更新等）会「污染」下一个 turn 的事件流。eventGate 的做法是：让 stream 自然结束，但用 gate 过滤掉超时后的所有事件。这比 abort + catch 更优雅，也避免了强制中断可能带来的资源清理问题。

4.2 failover：两次机会，不是无限重试

// attempt < 2：最多两次机会
// 触发条件（三者同时满足）：
// 1. attempt === 0（首次尝试）
// 2. sawTurnOutput === false（没有任何输出）
// 3. isRecoverableAcpxExitError()：匹配 "acpx exited with code/signal"

if (shouldRetry) {
  retryFreshHandle = true  // 换新的 handle（重新建连，不复用旧连接）
  continue
}
// 第二次失败 → 直接抛出，不再重试
工程决策

「两次机会」而不是指数退避。AI Agent 的对话是有时间敏感性的——用户在等。如果 failover 像 HTTP 请求一样做指数退避（几秒、几十秒……），用户体验会很差。两次机会是一个明确的上限：一次是正常情况，一次是临时故障恢复，超过两次基本说明是系统性问题，需要人工介入，不应该继续自动重试。

4.3 taskProgressSummary：后台任务的实时追踪

对于 background task 模式（mode === 'prompt'），每个 text_delta 都追加到 progressSummary，turn 结束时通过 resolveBackgroundTaskTerminalResult() 提取结构化的 terminalOutcome。这让后台任务的执行过程完全可观测，不是「发出去等结果」的黑箱。

五记忆体系：向量数据库 + QMD 语义索引

OpenClaw 选择了 LanceDB + 向量嵌入来实现跨 session 的记忆，而不是 Claude Code 的 Markdown 文件方案。这是两个框架最大的架构差异之一。

5.2 auto 选择的负向约束

// auto 模式：REMOTE_EMBEDDING_PROVIDER_IDS
const REMOTE = ["openai", "gemini", "voyage", "mistral"]
// 注意：Ollama 故意不在这个列表里
// 原因：不能假设用户有 Ollama（部署在服务器上可能根本没有）

// canAutoSelectLocal() 的判断逻辑
function canAutoSelectLocal(modelPath: string): boolean {
  if (!modelPath) return false
  if (modelPath.startsWith('hf:')) return false     // HuggingFace 需要下载
  if (modelPath.startsWith('http:')) return false   // 远程 URL
  return fs.existsSync(modelPath)  // 只有文件实际存在才可 auto 选择
}
工程决策

负向约束（exclusion list）比正向约束（inclusion list）更安全。auto 选择的原则是「只选我确定存在的资源」，而不是「选我觉得可能有的资源」。这个原则防止了「用户配置了 Ollama 但没有启动，auto 选到 Ollama 然后失败」的场景——在生产环境中，这种「悄悄失败」是最难调试的问题。

5.3 Gemini 向量的特殊能力

Gemini 的 embedding API 比 OpenAI 多了两个维度：

  • outputDimensionality：可选 768 / 1536 / 3072 维——维度越高，语义区分能力越强，但存储和检索成本也越高
  • taskType：RETRIEVAL_DOCUMENT / RETRIEVAL_QUERY / CLASSIFICATION 等——告诉模型「这个向量用来做什么」，让模型在生成 embedding 时考虑下游任务的需求

这个「任务感知」特性意味着同一段文本，用于「被检索」和用于「检索别人」会生成不同的 embedding，检索精度理论上比通用 embedding 更高。

6.2 安装安全扫描

安装 skill 时，OpenClaw 会对代码进行安全扫描：

  • critical findings → 阻断安装（security_scan_blocked）
  • warn findings → 告警不阻断（展示给用户决策）
  • scan failure → 阻断安装（security_scan_failed，扫描工具本身出错也不允许跳过）
工程决策

scan failure 阻断，而不是跳过。如果扫描工具报错了就允许安装，等于「关掉报警继续施工」。阻断的代价是用户需要调查为什么扫描失败，但这比允许未经检验的代码运行要安全得多。

7.3 exec-approvals：三级安全 × 三级询问

// ExecSecurity（安全策略）

// ExecApprovalDecision：每个 approval request 可限制哪些 decision 可用
// 防止用户对危险命令选择 allow-always

// DEFAULT_EXEC_APPROVAL_TIMEOUT_MS = 1,800,000（30分钟）
// 用户有充分的时间查看和决定
工程决策

normalizeExecApprovals 三步防腐。allowlist 数据在持久化时可能因为用户手动编辑、版本迁移等原因损坏。三步处理：① coerceAllowlistEntries（兼容旧格式）→ ② ensureAllowlistIds（补 UUID）→ ③ stripAllowlistCommandText（持久化时移除明文命令，只保留 pattern）。这三步的顺序不能颠倒，且每步都是幂等的。

startup catchup 防雪崩

// 场景：gateway 宕机 2 小时，重启后有 50 个 cron job 错过了
// 不加保护：50 个 job 同时执行，瞬间打满 API 配额

// OpenClaw 的做法：有上界 + 错峰
const missedJobs = findMissedJobs(since: lastRestartTime)
const toRun = missedJobs.slice(0, MAX_MISSED_JOBS_PER_RESTART)  // 最多 5 个

for (const [i, job] of toRun.entries()) {
  await sleep(i * DEFAULT_MISSED_JOB_STAGGER_MS)  // 每个延迟 5s
  executeJob(job)
}
// 结果：最多 5 个，25s 内全部完成，不会并发爆发
工程洞察

「有上界的补跑」是关键设计。如果系统宕机时间很长，补跑所有错过的 job 可能会造成比宕机本身更大的影响（雪崩效应）。OpenClaw 选择了「最多补 5 个，多的直接跳过」，这是一个明确的业务权衡：宁愿丢弃一些历史 job，也不能在重启时再次造成系统过载。

九多账号轮转：真正的 Round-Robin

排序算法的精妙之处

// 排序优先级（降序）：
// 1. preferredProfile → 用户显式指定，始终第一
// 2. 无 cooldown 的 profile：
//    按类型：oauth > token > api_key
//    按 lastUsed：最旧的先用（！这是关键）
// 3. 有 cooldown 的 profile：按到期时间升序

// 关键注释（原文保留）：
// lastGood is NOT prioritized - that would defeat round-robin
// round-robin 的意义是均匀分摊负载
// 不是「总用上次成功的」——那叫「总用同一个」
工程决策

lastGood 不参与排序是很多工程师容易犯的直觉错误：「上次成功的就是最好的，应该优先用」。但在负载均衡场景下，这等于「把所有流量打到一个 endpoint」。OpenClaw 选择的是真正的 round-robin：不管上次谁成功，这次用「最久没用过的」，确保所有 profile 的使用量趋向均衡。

工程洞察

配置漂移是生产环境中最常见的「安静失败」来源。如果不做这个修复，用户账号迁移后 OpenClaw 会静默使用错误的 profile 或者报错，而用户不知道为什么。自动修复虽然不是「正确答案」，但比「静默失败」好得多——至少系统还在工作。

十Sub-agent 持久化：孤儿恢复状态机

三层持久化设计

// 三层存储，从快到慢：
// in-memory Map → persistSubagentRunsToDisk (JSON) → orphan-recovery (重启后)

// 孤儿恢复决策树（重启后扫描所有 sub-agent runs）：
if (cleanupCompletedAt) return 'skip'  // 已清理

if (announceRetryCount >= MAX_RETRY)
  return finalizeResumedAnnounceGiveUp('retry-limit')  // 重试次数耗尽

if (timeSince(endedAt) > ANNOUNCE_EXPIRY_MS)
  return finalizeResumedAnnounceGiveUp('expiry')  // 太老了，不再通知

if (suppressAnnounceForSteerRestart)
  return 'skip'  // 被 steer 中断的，等 steer 替换，不通知

if (endedAt > 0)
  return startSubagentAnnounceCleanupFlow()  // 已结束，发通知

return waitForSubagentCompletion(runId, waitTimeoutMs)  // 还在运行
工程决策

suppressAnnounceForSteerRestart 防止双重通知。当用户通过 steer 命令替换正在运行的 sub-agent 时，旧的 run 被中断，新的 run 会接管。如果旧 run 在重启后还发通知，用户会收到「你刚刚取消的任务」的通知，体验很差。这个 flag 记录了「这个 run 是被主动中断的」，重启后跳过通知。

动态 import：懒加载孤儿恢复

// 孤儿恢复模块通过动态 import 延迟加载
// 效果：不影响 gateway 冷启动时间和内存

void import('./subagent-orphan-recovery.js')
  .then(({ scheduleOrphanRecovery }) => {
    scheduleOrphanRecovery(...)
  })
  .catch(() => {
    // best-effort：失败了也不影响正常运行
    // 只是这次重启不做孤儿恢复
  })

sweeper 节能

归档 sweeper 不会空转：

  • 只有当存在有 archiveAtMs 的 run 时才启动 sweeper
  • 最后一个需要归档的 run 清除后，自动 stopSweeper()
  • 不保留空转的定时器——Node.js 进程不会因为 sweeper 无法退出

十一工程模式总结

可直接借鉴的工程模式

Pattern 1
eventGate 防超时污染
turn 超时后关闭 gate，让 stream 自然结束但过滤所有事件。比 abort+catch 更优雅，避免强制中断的副作用。
Pattern 2
三源一致性反注入
从多个来源提取关键字段，任意两源不一致则拒绝。防止通过一个来源伪造信息绕过权限检查。
Pattern 3
有上界的补跑
重启后最多补跑 N 个错过的 cron job，超过的直接跳过。防止补跑导致的雪崩效应，是资源管理的重要约束。
Pattern 4
负向约束的 auto 选择
auto 选择模式主动排除「不确定存在」的资源（Ollama、hf: 路径）。只选「确定存在」的，而不是「可能有的」。
Pattern 5
pending 覆盖而非队列
流式更新用覆盖而非队列，配合 throttleMs 控制频率。永远发最新内容，避免历史积压。
Pattern 6
动态 import 懒加载
安全审计模块、孤儿恢复模块通过动态 import 延迟加载，不影响 gateway 冷启动。best-effort 失败静默处理。
Pattern 7
防腐三步
持久化数据读取时：coerce（兼容旧格式）→ ensureIds（补缺失字段）→ strip（去除不应持久化的内容）。每步幂等，顺序不可颠倒。
Pattern 8
变换语义 Hook
Hook 函数返回修改后的对象，而不是 exit code。支持「参数注入」、「内容过滤」等复杂变换，比门控语义（通过/拒绝）更灵活。

值得警惕的反模式

反模式
  • scan failure 跳过：安全扫描工具报错就继续安装，等于「关报警继续施工」
  • lastGood 优先排序：在负载均衡场景下，「总用上次成功的」= 把所有流量打到一个 endpoint
  • 补跑不设上界：重启后无限补跑历史 job，可能比宕机本身造成更大破坏
  • auto 选择假设资源存在：Ollama 可能没启动，HuggingFace 模型可能没下载，不能假设

十二工程哲学：三个关键词

可观测性优先

OpenClaw 在多个地方内置了观测点：SessionActorQueue 暴露 getPendingCountForSession() 供监控系统接入；RuntimeCache.snapshot() 返回含 idleMs 的快照；ACP manager 内置 turnLatencyStats 和 errorCountsByCode。这些不是事后加的，而是从设计之初就内置的。

资源有界

几乎每个可能无限增长的资源都有明确的上界：maxTrackedKeys = 2048（防抖 Map）、MAX_MISSED_JOBS_PER_RESTART = 5（补跑上限）、maxConcurrentRuns（cron 并发）、announceRetryCount >= MAX_RETRY（孤儿恢复重试上限）。这不是「过度防御」，而是「在最坏情况下系统仍然可预测」。

最小惊讶原则

系统行为应该符合用户预期。suppressAnnounceForSteerRestart 防止收到「你刚才取消的任务」的通知；normalizeExecApprovals 在读取时自动修复损坏数据，而不是报错；allBaseProfilesMissing 自动 fallback 而不是静默失败。这些设计的共同点是：用户的操作应该产生用户期望的结果，而不是让用户惊讶。

把并发问题变成串行问题；把资源问题变成有界问题；把观测问题变成设计问题。
不要让系统做用户没有预期的事，也不要让系统在用户不知情的情况下失败。
