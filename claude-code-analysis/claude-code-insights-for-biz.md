# 读完 Claude Code 源码，我找到了 5 个适用于所有产品的工程智慧

> 这不是技术文章，是写给产品经理、设计师、和对工程感兴趣的业务同学的。
> 背景：我们花三天读完了 Claude Code 约 6 万行源码，提炼出 20 个核心设计决策。
> 这篇文章从中选出 5 个，翻译成人话。

---

## 写在前面：Claude Code 是怎么运行的

Claude Code 是 Anthropic 推出的 AI 编程助手，可以直接在终端里帮你写代码、改 bug、分析项目。

它内部有一个「主循环」：不断地接收你的消息 → 调用 Claude API → 执行命令 → 再接收反馈。这个循环每天在全球数十万用户的电脑上运行数百万次。

这意味着它的每个设计决策，都会被放大到数百万倍。每个 100ms 的延迟，乘以数百万次，就是数十年的用户等待时间。每个多余的 API 调用，乘以数百万次，就是真实的金钱成本。

所以它的工程师们在每个细节上都问了一个问题：**「这个操作能不能更便宜？能不能让用户等更少？能不能让系统更可信？」**

以下是 5 个我觉得最值得借鉴的答案。

---

## 第一课：把用户的「等待时间」变成「计算时间」

**故事：API 密钥的预取**

Claude Code 启动时需要读取 API 密钥，这个操作需要 200-400ms。如果就在用户等着用的时候才去读，用户就会看到一段莫名其妙的空白。

Claude Code 的做法是：**在 app 刚启动、用户还在看欢迎界面的时候，就悄悄开始读了**。等用户输完第一条消息按下回车，密钥早就准备好了。

```
用户视角：我点了按钮 → 立刻有反应
实际发生：app 启动 → 后台悄悄读密钥 → 用户输入 → 密钥已就绪
```

**更大的模式：用户阅读界面的时间 = 系统计算的时间**

这个原则在 Claude Code 里被反复使用：
- 用户看「是否允许这个命令？」的提示时 → 后台已经在问 AI「这个命令安全吗？」
- 用户输入下一条消息时 → 后台已经在整理「上这轮对话的关键记忆」
- 用户阅读上一条回复时 → 下一次 API 调用的「缓存」已经预热好了

**给业务的启示：**

在你的产品里，找那些「用户必须等待」的操作，想想能不能把它分成「用户在做 A 的时候，系统悄悄做 B」。比如：
- 用户填表的时候，后台验证前几个字段
- 用户浏览列表的时候，后台预加载下一页
- 用户看确认弹窗的时候，后台准备提交需要的数据

**延迟不是必然的，往往只是「等待时机的顺序没设计好」。**

## 第二课：把昂贵操作做成「最后手段」

**故事：三级压缩体系**

Claude Code 每隔一段时间需要「压缩」对话历史（因为 AI 处理的内容有上限）。压缩有很多种方法，从便宜到贵排列：

| 方法 | 成本 | 说明 |
|------|------|------|
| 时间压缩 | 几乎为零 | 缓存过期了就直接清空 |
| 选择性保留 | 低 | 保留最重要的内容，丢弃其余 |
| 智能裁剪 | 低 | 按预算削减不重要的内容 |
| AI 摘要 | 最贵 | 让 Claude 读完所有历史，写一个摘要 |

Claude Code **只有在前三种都行不通时，才会用最贵的那种**。而且即使用最贵的，也是 85% 快满的时候才触发，不是 95%（那样来不及了）也不是 70%（那样太浪费了）。

**更大的模式：从便宜到贵，一层一层往下走**

这个「成本梯度」设计在 Claude Code 里很多地方都有：
- 检查权限：先查「有没有现成规则」→ 没有才问「快速 AI 检查」→ 还不确定才「仔细 AI 推理」
- 记忆整理：先「时间没到不整理」→ 然后「新 session 不多不整理」→ 才「用文件锁保证不重复整理」
- 任务取消：先「发结束信号」→ 等超时才「强制杀进程」

**给业务的启示：**

在你的产品里，找那些「贵」的操作（API 调用、数据库写入、邮件通知、人工介入……），想想能不能在它们前面加几道「便宜的门」：

> 先检查缓存 → 再查本地规则 → 再调用外部服务

用户点了「发送验证码」按钮：先检查 5 分钟内有没有发过（本地记录，极快）→ 再检查这个手机号有没有被封（缓存查询，快）→ 才真正发短信（贵）。

**大多数「贵」的操作，其实大多数时候根本不需要执行。**

## 第三课：让自动化系统知道自己的边界

**故事：Denial Tracking 防死锁**

Claude Code 有一个 `auto` 权限模式：用 AI 自动决定「这个命令要不要执行」，不打扰用户。听起来很美——直到 AI 分类器连续误判，把本来应该执行的命令反复拒绝，导致 Claude Code 什么都做不了，陷入「死锁」。

为了防止这种情况，代码里有一个简单但有效的计数器：

```
连续拒绝 ≥ 3 次
OR
总拒绝 ≥ 20 次

→ 停止自动决策，强制弹出「请人类确认」
```

**不是信任 AI 会永远正确，而是知道什么时候 AI 可能出了问题，需要人类接管。**

还有另一个维度：`bypassPermissions`（完全跳过所有权限确认）这个模式，有两个独立的「开关」，可以在不更新代码的情况下远程关闭。一旦发现这个模式被滥用或者出了问题，可以在几分钟内对全部用户生效，不需要发版本、不需要强制更新。

**更大的模式：自动化系统需要「self-awareness 限流」**

自动化越强大，失控的代价越大。好的自动化系统应该知道：
- 我在什么情况下可能是错的？
- 出错的时候，我怎么告诉人类「我需要帮助」？
- 如果我一直出错，有没有一个硬性的「停下来等人类」机制？

**给业务的启示：**

产品里的任何自动化决策（推荐算法、风控规则、自动操作……），都应该有：
1. **异常计数器**：连续失败/异常多少次就暂停？
2. **人工兜底路径**：自动化失败时，用户能不能回到「人工确认」？
3. **远程开关**：出问题时能不能不发版就关掉？

**「自动化」和「人类控制」不是对立的，好的系统是「自动化优先，人类兜底」。**

## 第四课：「成功」和「失败」都值得记录

**故事：feedback 记忆类型的设计**

Claude Code 有一个跨 session 的长期记忆系统，会记住用户的偏好和使用习惯。记忆的类型里有一个叫 `feedback`，用来记录用户的反馈。

有意思的是：**它同时记录纠正（「别这么做」）和确认（「就是这样，继续」）。**

大多数「学习」系统只学失败：被纠正了就记住「不要做 X」。但它们忘记了：**「对了」也需要被记住**——否则下次遇到类似情况，系统会忘记它已经找到了正确做法，重新犹豫，或者因为过于保守而不敢用。

**用户说「对对对就是这样」，这也是珍贵的信号，不应该被丢掉。**

**更大的模式：正反馈和负反馈都是数据**

这个洞察可以很直接地翻译成产品决策：

- 用户在你的推荐结果里点了哪些？（正反馈）
- 用户忽略了哪些？（负反馈）
- 用户专门回来找了哪些内容？（强正反馈）
- 用户取消订阅是什么时候发生的？（强负反馈）

很多产品只优化「减少投诉」，忘记了「放大做对的事」。两者同样重要。

**给业务的启示：**

想想你的产品收集了哪些用户反馈信号。有没有「用户说好」的信号被系统性地忽视了？

常见的被忽视的正向信号：
- 用户把某个内容收藏/分享了（不只是点了赞）
- 用户把某个功能设为默认（说明它真的好用）
- 用户在某个页面停留特别长（在认真看，不是迷路了）
- 用户从客服投诉升级到「这个产品救了我」（这些故事同样重要）

**你不记录的东西，算法也学不到。你不重视的反馈，产品也不会改进。**

## 第五课：在「自动」和「人工」之间，建一个渐进台阶

**故事：七种权限模式的设计**

Claude Code 不是「要么全自动，要么全手动」，而是设计了七种权限模式，形成一个连续的「信任光谱」：

```
plan（只读，不执行任何操作）
↓
default（危险操作弹确认）
↓
acceptEdits（文件编辑自动接受，命令仍然询问）
↓
auto（AI 分类器自动决策，出问题时兜底人工）
↓
dontAsk（静默拒绝所有询问，不打扰用户）
↓
bypassPermissions（完全信任，跳过所有确认）
```

用户可以在任何时候移动这个滑块。**「我刚开始用，想看清楚发生了什么」→ plan；「我相信这个工具了」→ acceptEdits；「我要完全放手让它跑」→ bypassPermissions。**

这不只是「功能丰富」，而是**对不同信任程度、不同使用场景的用户，提供了适当的控制感**。

**更大的模式：自动化不是非 0 即 1**

很多产品在「自动化」的设计上走了极端：要么「全自动，用户没有控制感」，要么「什么都要用户确认，麻烦死了」。

但用户的信任是逐渐建立的：
- 第一次用：我想看清楚它在做什么
- 用了一段时间：我信任它做这类事情，但新类型的操作还想确认
- 深度用户：我完全信任它，别来打扰我

对应的产品设计应该是：**让用户可以「渐进授权」，而不是一开始就「全给或全不给」**。

**给业务的启示：**

想想你的产品里有哪些「权限类型」：通知权限、数据权限、操作权限……

有没有可能设计一个「渐进台阶」：
- 第一次：「我来帮你，但我每步都告诉你我在做什么」
- 几次之后：「这类事情我可以直接做了，对吗？」
- 深度用户：「全权委托给我，你去做别的」

**用户的信任是赚来的，不是一开始就应得的。给他们控制权，他们才会给你授权。**

---

## 总结：一句话的工程哲学

这五课，用一句话概括：

> **把昂贵的事变便宜；把等待的时间变计算；把自动化的边界画清楚；把对的和错的都记住；让用户用信任换授权。**

这不只是软件工程的智慧，也是所有「代理」（AI、产品功能、服务团队）与用户建立关系的底层逻辑。
